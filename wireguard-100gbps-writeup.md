# Scaling WireGuard with a multi-tunnel fabric — measured results

> **Status: MEASURED** (three live runs on **2026-06-19**, two `i8ge.48xlarge` Spot nodes in
> `us-west-2d`, generated by `report/` from `results/`). Nothing here is fabricated; every
> value traces to a `datapoint.json` with attribution from counter/CPU deltas on **both** nodes.
>
> **Headline:** the multi-tunnel fabric scales **near-linearly from 7.9 Gbps (N=1) to
> ~48–55 Gbps (N=8)**, then the *unpinned* curve plateaus at **~57–60 Gbps** — bound by
> **aggregate receive-side CPU** (NAPI poll + ChaCha20 decrypt kworkers + ksoftirqd + the
> userspace receiver), *not* by any AWS allowance and *not* by the ~180 Gbps network wall
> (raw ENA hit 208 Gbps with no WG).
>
> **The bottleneck is CPU placement, not a hard wall.** Run 2 instrumented per-core CPU and
> found the receiver spends **~57 core-equivalents across ~31 cores at N=32** — broadly
> distributed, *not* one pegged core (the earlier "single core at 99%" reading was one
> marginally-hotter core atop wide load). Simply **pinning the receiver's worker threads to
> distinct cores lifted N=32 from 60 → 77 Gbps (+29%)** at the *same* CPU cost
> (1.04 → 1.35 Gbps/core-equiv). 100 Gbps was still not reached, but the path there is
> clearly per-core efficiency / core-placement, not the network.
>
> **NUMA placement is the next big lever.** `i8ge.48xlarge` is 2 NUMA nodes with the NIC on
> node 1; local memory BW is 381 GB/s vs 69 remote (90% penalty). After fixing a NUMA-blind
> IRQ-pinning bug, putting the **whole receive pipeline (IRQ + softirq + decrypt + userspace)
> on the NIC-local node** reached **89.5 Gbps at N=32** — the project's best, within reach of
> 100 Gbps — vs ~60 unpinned and 55 with userspace on the wrong node. The constraint remains
> per-core receive-CPU efficiency; correct NUMA placement buys ~50% over naive.

## Thesis (as stated going in)

A single WireGuard tunnel can't reach 100 Gbps on an `i8ge.48xlarge`, because every
bottleneck is keyed on the **flow** — and a WireGuard interface is one UDP 5-tuple:

1. **Per-flow bandwidth cap** — one 5-tuple is capped (5 / 10 / 25 Gbps).
2. **Single RX queue** — one 5-tuple RSS-hashes to one ENA queue → one core's softirq.
3. **Per-peer crypto ordering** — one tunnel's ChaCha20-Poly1305 is one serial pipeline.

N tunnels = N flows = N RX queues = N crypto pipelines, so throughput should scale
near-linearly with N until a **global** ceiling binds. The prediction was that the global
ceiling would be the ~180 Gbps VPC allowance. **The measured global ceiling turned out to be
aggregate receive-side CPU/crypto at ~60 Gbps instead** — see Findings.

## Rig

- 2 × `i8ge.48xlarge` — Graviton4, **192 vCPU**, 1536 GiB DDR5-5600, single network card,
  16 × 6.8 TB local Nitro NVMe instance store. Spot, `us-west-2d`, cluster placement group.
- Ubuntu 24.04 arm64, **kernel 6.17.0-1017-aws**, in-tree WireGuard. ChaCha20-Poly1305.
- Jumbo: **9001** MTU on the ENA (`ens146`), **8921** on the 32 wg interfaces.

## Real workload — NVMe→NVMe over nvme-tcp (native multipath)

One nvme-tcp path per tunnel against a single target subsystem (16 namespaces), round-robin
iopolicy, so each I/O command rides a distinct tunnel/flow.

| mode | N | NVMe GB/s | equiv. Gbps | binding limit (measured) |
|---|---|-----------|-------------|--------------------------|
| read | 8 | 6.2 | 49.8 | linear region |
| read | 16 | **8.9** | 71.3 | CPU / crypto |
| read | 32 | 8.4 | 67.1 | CPU / crypto |
| write | 8 | 6.4 | 51.2 | linear region |
| write | 16 | 7.4 | 59.0 | linear region |
| write | 32 | 8.3 | 66.7 | CPU / crypto |

The real storage path **tracks the synthetic sweep** (~50–71 Gbps equivalent, same CPU/crypto
plateau). Peak **8.9 GB/s read** is far below the 52 GB/s local NVMe ceiling and the 25.9 GB/s
write ceiling — so the limiter is the encrypted network/crypto path, exactly as the iperf
sweep showed, not the disks.

### NVMe near:far NUMA placement (run 4)

Given the 90% remote-memory penalty, does it matter whether the target drives are on the
NIC-local node (no hop) or the far node (extra cross-complex DMA)? Measured at N=32, exporting
only node-1 (near), only node-0 (far), or all 16 (balanced):

| target drives | read GB/s | write GB/s |
|---|---|---|
| 8× near (node 1, NIC-local) | **8.3** | 7.4 |
| 8× far (node 0) | 7.5 | 7.9 |
| 16 balanced (both nodes) | 7.2 | 7.8 |

**Storage placement barely matters here** — near vs far differs by ~10% on read, with near
slightly ahead on read and far marginally ahead on write (within run-to-run noise).
The reason: the workload (~9 GB/s ≈ 70 Gbps) is **CPU/network-bound at the same ~70 Gbps
plateau**, far below the 69 GB/s *remote* memory ceiling, so the cross-NUMA hop has ample
headroom and barely bites. The only real signal is **balanced read dropping to 6.9 GB/s** —
spreading across 16 drives (vs 8) added mild contention. Practical takeaway: the near:far drive
balance is **not a useful lever at this throughput**; it would only start to matter if storage
bandwidth approached the ~69 GB/s remote-memory ceiling, and CPU binds long before that. Keep
it simple — NIC-local drives, since "near" is marginally best and never worse.

## Findings

1. **The fabric scales near-linearly up to N≈8**, then plateaus. Placement: 7.9 → 15.7 → 29.1
   → 48.0 Gbps for N = 1, 2, 4, 8 (≈2× per doubling — the thesis's core mechanism is real).
2. **N=1 is single-RX-queue bound** (softirq concentrated on one core: top-core share ~0.78),
   exactly the per-flow serialization the thesis predicts. Adding tunnels relieves it.
3. **The unpinned plateau (~57–60 Gbps) is aggregate receive-side CPU, not the network and
   not one core.** No `bw_*`/`pps`/`conntrack` allowance fired at any N on either node. Run 2's
   per-core instrumentation shows the receiver spending **~57 core-equivalents across ~31
   cores >50% busy** at N=32 — broad, distributed load. The hot threads are exactly the
   encrypted-receive pipeline: `napi/wgN-0` (NAPI poll), `kworker/*-wg-crypt-wgN` (ChaCha20
   decrypt workqueues), `ksoftirqd/*`, and the userspace `iperf3` receivers. The earlier
   single-run reading of "one core at 99%" was one marginally-hotter core atop that wide load,
   not a serial bottleneck.
4. **The receiver costs ~4× the sender.** At N=32: receiver **57 core-equivalents** vs sender
   **15**. Decrypt + receive dominates; the receiver is the binding side, as predicted.
5. **Per-core efficiency degrades with N** (unpinned): 2.53 → 2.13 → 1.91 → 1.31 →
   **1.04 Gbps/core-equiv** for N = 1, 4, 8, 16, 32. Each added tunnel costs progressively
   more CPU per Gbps — softirq/scheduling/cache overhead from spreading the receive path ever
   wider. This diminishing return, not a hard wall, is what flattens the curve.
6. **Pinning the receive workers recovers a large chunk of that loss.** Pinning each tunnel's
   `iperf3` server to a distinct core (`pin-workers.sh`) lifted the curve substantially at
   high N:

   | N | unpinned Gbps | pinned Gbps | Δ |
   |---|---------------|-------------|---|
   | 8 | 48.0 | 42.2 | −12% |
   | 16 | 55.3 | 58.6 | +6% |
   | 24 | 55.8 | 69.2 | +24% |
   | 32 | 59.9 | **77.2** | **+29%** |

   At N=32 this used the **same ~57 core-equivalents** but delivered 77 vs 60 Gbps —
   efficiency **1.04 → 1.35 Gbps/core (+30%)**. Same CPU budget, more throughput: the gain is
   pure cache-locality / reduced task migration. (At low N pinning slightly *hurts* — it
   over-constrains when there is spare capacity to schedule freely.)
7. **100 Gbps was still not reached** (best: 77 Gbps, pinned, N=32) — but the binding
   constraint is decisively **per-core efficiency / CPU placement**, not the ~180 Gbps VPC
   wall (raw ENA hit 208 Gbps with no WG).
8. **ENA Express made no measurable difference**, consistent with SRD barely engaging (tiny
   `ena_srd_tx_pkts` vs millions eligible) and with the ceiling being CPU-bound.
9. **The real NVMe→NVMe workload (run 1) confirmed the synthetic curve** — same shape, same
   ceiling, well under the local storage ceilings.

### Honest caveats

- **Pinning was userspace-only.** `pin-workers.sh` pins the `iperf3` servers; it does **not**
  pin the kernel's wg-crypt kworkers or ksoftirqd (kernel-controlled). The +29% came from
  pinning userspace alone — pinning the crypto path too (via RPS/RFS / XPS / IRQ-to-decrypt
  affinity) is the obvious next lever and might close more of the gap to 100 Gbps.
- **`busy_cores` (cores >90%) under-counts** when work spreads thinly; `busy_core_equiv`
  (core-equivalents) is the metric that actually answers "how many cores' worth of work" and
  is what the findings above rely on.
- **Short windows (DUR=15–20 s).** Steady-state for the sweep, but SRD's adaptive ramp wants
  longer; a multi-minute `ena_express` run might engage SRD materially.
- **Pinned run was placement-mode only** (one A/B). A pinned `ena_express` and a finer N grid
  near the knee would sharpen the picture.

### What would push past 77 Gbps (hypotheses, partly tested)

The lever is per-core receive efficiency, now demonstrated by the pinning result:

- **NUMA-aware placement** — if the instance presents >1 NUMA node, pin the decrypt path and
  workers to the node the ENA NIC is local to, so packets don't cross the interconnect. See
  the dedicated note below; `detect-numa.sh` + `NODE=<n> pin-workers.sh` test this directly.
- **Pin the kernel decrypt path too** — RPS/RFS to steer each tunnel's RX to a fixed core,
  XPS for TX, and align the wg-crypt kworker affinity with the RX core (cache-warm decrypt).
- **Pipeline the stages** (read→encrypt→network→decrypt→write) so a flow's decrypt, TCP
  delivery, and disk write run on different cores rather than contending — see the dedicated
  note below; this is the most promising untested direction.
- The network (208 Gbps) and storage (52 GB/s read) have ample headroom; only the
  encrypted-receive CPU pipeline binds.

### NUMA placement — MEASURED, and the result is counterintuitive

`i8ge.48xlarge` **presents 2 NUMA nodes** (node 0 = cores 0-95, node 1 = cores 96-191), and
the guest *does* expose device affinity: the **ENA NIC is on node 1** (`numa_node=1`), and the
16 instance-store NVMe split 8/8 across both nodes (`detect-numa.sh` → `results/numa.json`).

So I ran the decisive A/B on the receiver at N=8/16/24/32: pin all userspace workers to the
**NIC-local** node (`NODE=1`) vs the **NIC-remote** node (`NODE=0`):

| N | `NODE=1` NIC-local | `NODE=0` NIC-remote |
|---|--------------------|---------------------|
| 8 | 55.9 | 53.3 |
| 16 | 57.3 | 61.4 |
| 24 | 57.0 | 70.8 |
| 32 | **56.3** | **75.7** |

In this configuration the NIC-*remote* node was faster — but **read the confound before
drawing the obvious conclusion.** Instrumentation (N=32): NIC-local pinning smeared work
across **55 cores >50% busy** at 0.97 Gbps/core-equiv; NIC-remote concentrated it in **30
cores** at 1.39 Gbps/core-equiv.

**Confound (important):** `node-setup.sh`, as run for these sweeps, pinned the 32 ENA RX IRQs
to cores **0–31 — all on node 0, the NIC-*remote* node** (its round-robin started at core 0
and was NUMA-blind; the NIC is on node 1). So the RX softirq/NAPI processing was on node 0
the whole time. That means the A/B actually compared:

- **`NODE=0`:** userspace receivers **co-located with the RX-softirq cores** (both node 0) →
  76 Gbps, work concentrated, cache-warm hand-off from softirq to the userspace socket.
- **`NODE=1`:** userspace receivers **split from the RX-softirq cores** (userspace node 1,
  softirq node 0) → 56 Gbps, cross-node hand-off on every packet.

So the honest finding from run 3 was **"co-locate userspace receive with the RX-softirq
cores," not "avoid the NIC-local node"** — the IRQs never actually ran on the NIC-local node,
so the textbook layout was untested.

**Run 4 fixed the IRQ pinning and retested** (IRQs now verified on NIC-local cores 96–191;
softirq/ksoftirqd followed them there). The A/B *flipped*, confirming the textbook layout and
the original NUMA intuition:

| N=32, IRQs NIC-local (node 1) | Gbps | Gbps/core-equiv |
|---|---|---|
| userspace `NODE=1` (all NIC-local) | **89.5** | 1.42 |
| userspace unpinned | 59.8 | 0.99 |
| userspace `NODE=0` (remote from softirq) | 55.4 | 0.93 |

(Re-run values; the all-NIC-local layout reached **89.5 Gbps** — the project's best, with the
receiver at 62.9 busy core-equivalents / 100% peak core. The receive threads at N=32 are
`ksoftirqd` + `wg-crypt` kworkers on node-1 cores 96–191, confirming the whole pipeline ran
NIC-local as intended.)

**Conclusion (measured, not inferred): keep the whole receive pipeline — RX IRQ, NAPI/softirq,
decrypt, and userspace — on the NIC-local node.** This took the fabric to **89.5 Gbps** at
N=32 — within striking distance of 100 Gbps and far above the unpinned ~60. Run 3's "remote
wins" was purely an artifact of NUMA-blind IRQ placement (`node-setup.sh` pinned IRQs to cores
0–31 = the remote node); once corrected, NIC-local co-location is best, as hardware locality
predicts.

**Per-NUMA memory bandwidth** (run 4, `measure-membw-numa.sh`): **local ≈ 381 GB/s, remote ≈
69 GB/s — a 90% / 5.5× penalty** (far steeper than the firmware `node distances` 10:20 implies;
the old all-192-core "5551 GB/s" was measuring cache, a now-fixed bug — real aggregate DRAM is
~762 GB/s). This is *why* crossing the complex hurts, and why NIC-local co-location wins.

### Note: pipelining the read→encrypt→network→decrypt→write path

The current setup does **not** explicitly pipeline stages across cores — affinity is per-flow
(one tunnel's whole receive chain tends to land together), and the +29% pinning win shows how
sensitive throughput is to placement. A genuine pipeline — decrypt on core X, TCP/socket
delivery on core Y, disk write on core Z, handed off via per-stage queues — could keep more
cores productive per flow and raise the per-core-equivalent efficiency further. This is
recorded as the top follow-up; it is **not yet measured**.

### Run log

- **Run 1** (initial): placement + ena_express sweeps + NVMe→NVMe over nvme-tcp; surfaced six
  real-hardware harness bugs (now fixed). Established the ~60 Gbps unpinned plateau and that
  no AWS allowance binds.
- **Run 2** (instrumented): per-core histogram + `top_threads` + the pinning A/B. Resolved
  "how many cores / which threads" (≈57 core-equivalents across ~31 cores; napi + wg-crypt
  kworkers + ksoftirqd + iperf3) and demonstrated the **60 → 77 Gbps** pinning gain.
- **Run 3** (NUMA probe): classified the instance (2 nodes, NIC on node 1) and ran the
  userspace A/B — which *appeared* to favor the NIC-remote node, but exposed the NUMA-blind
  IRQ-pinning confound (IRQs were on the remote node).
- **Run 4** (NUMA, corrected): fixed IRQ pinning to NIC-local; the A/B flipped to confirm
  **all-NIC-local is best**. Measured per-NUMA memory bandwidth (381 local / 69 remote GB/s)
  and the NVMe near:far ratio (placement barely matters at this CPU-bound rate). A merge bug
  lost most of run 4's raw `datapoint.json` before commit; **run 4b re-ran the full experiment
  and captured it correctly** — the all-NIC-local layout reached **89.5 Gbps** (project best).
