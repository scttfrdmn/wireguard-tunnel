# Scaling WireGuard with a multi-tunnel fabric — measured results

> **Status: MEASURED.** Numbers below are from a live run on **2026-06-19**, two
> `i8ge.48xlarge` Spot nodes in `us-west-2d`, generated by `report/` from `results/`.
> Per the methodology rule, nothing here is fabricated; every value traces to a
> `datapoint.json` and the attribution comes from counter/CPU deltas on **both** nodes.
>
> **Headline:** the multi-tunnel fabric scaled **near-linearly from 7.9 Gbps (N=1) to
> ~55 Gbps (N=8)**, then **plateaued at ~57–60 Gbps** — bound by **receive-side CPU/crypto**,
> *not* by any AWS bandwidth/PPS allowance and *not* by the ~180 Gbps network wall. The
> 100 Gbps target was **not** reached on this rig with this configuration. That is the
> honest result, and the attribution below shows exactly why.

## Thesis (as stated going in)

A single WireGuard tunnel can't reach 100 Gbps on an `i8ge.48xlarge`, because every
bottleneck is keyed on the **flow** — and a WireGuard interface is one UDP 5-tuple:

1. **Per-flow bandwidth cap** — one 5-tuple is capped (5 / 10 / 25 Gbps).
2. **Single RX queue** — one 5-tuple RSS-hashes to one ENA queue → one core's softirq.
3. **Per-peer crypto ordering** — one tunnel's ChaCha20-Poly1305 is one serial pipeline.

N tunnels = N flows = N RX queues = N crypto pipelines, so throughput should scale
near-linearly with N until a **global** ceiling binds. The prediction was that the global
ceiling would be the ~180 Gbps VPC allowance. **The measured global ceiling turned out to be
aggregate receive-side CPU/crypto at ~60 Gbps instead** — see Findings.

## Rig

- 2 × `i8ge.48xlarge` — Graviton4, **192 vCPU**, 1536 GiB DDR5-5600, single network card,
  16 × 6.8 TB local Nitro NVMe instance store. Spot, `us-west-2d`, cluster placement group.
- Ubuntu 24.04 arm64, **kernel 6.17.0-1017-aws**, in-tree WireGuard. ChaCha20-Poly1305.
- Jumbo: **9001** MTU on the ENA (`ens146`), **8921** on the 32 wg interfaces.

| Ceiling | Tool | Measured | Notes |

| Raw ENA throughput (no WG, 32 streams) | `measure-baseline.sh` | **208.1 Gbps** | above the nominal 180 — burst allowance over a 30 s window |
| Raw ENA packet rate | `measure-baseline.sh` | **2.90 M pps** | at jumbo MTU |

| NVMe striped sequential **write** (16 dev) | `measure-nvme.sh` | **25.9 GB/s** | the sink-side storage limiter |

The raw ENA path reaching **208 Gbps with no WireGuard** is itself notable — it proves the
network fabric is *not* the constraint at 60 Gbps. The bottleneck the sweep hits is above WG.

## Real workload — NVMe→NVMe over nvme-tcp (native multipath)

One nvme-tcp path per tunnel against a single target subsystem (16 namespaces), round-robin
iopolicy, so each I/O command rides a distinct tunnel/flow.

| mode | N | NVMe GB/s | equiv. Gbps | binding limit (measured) |
|---|---|-----------|-------------|--------------------------|
| read | 8 | 6.2 | 49.8 | linear region |
| read | 16 | **8.9** | 71.3 | CPU / crypto |
| read | 32 | 8.4 | 67.1 | CPU / crypto |
| write | 8 | 6.4 | 51.2 | linear region |
| write | 16 | 7.4 | 59.0 | linear region |
| write | 32 | 8.3 | 66.7 | CPU / crypto |

The real storage path **tracks the synthetic sweep** (~50–71 Gbps equivalent, same CPU/crypto
plateau). Peak **8.9 GB/s read** is far below the 52 GB/s local NVMe ceiling and the 25.9 GB/s
write ceiling — so the limiter is the encrypted network/crypto path, exactly as the iperf
sweep showed, not the disks.

## Findings

1. **The fabric scales near-linearly up to N≈8**, then plateaus. Placement: 7.9 → 15.8 → 30.9
   → 54.7 Gbps for N = 1, 2, 4, 8 (≈2× per doubling — the thesis's core mechanism is real).
2. **N=1 is single-RX-queue bound** (softirq concentrated on one core: top-core share 0.78),
   exactly the per-flow serialization the thesis predicts. Adding tunnels relieves it
   (share drops to 0.07 by N=32).
3. **The plateau (~57–60 Gbps) is CPU/crypto, not the network.** No `bw_*`/`pps`/`conntrack`
   allowance counter fired on either node at any N; meanwhile the **receiver** (decrypt side)
   shows `max_core_util` climbing to **99% (N=16) / 98% (N=32)**. The classifier attributes
   these rows to "CPU / crypto" on that measured signal.
4. **100 Gbps was not reached**, and the binding constraint was **not** the ~180 Gbps VPC
   wall — the raw ENA baseline hit 208 Gbps with no WG, proving headroom. The encrypted
   fabric tapped out ~3× lower, on the receive-side CPU/crypto path.
5. **ENA Express made no measurable difference** here, consistent with SRD barely engaging
   (see note) and with the ceiling being CPU-bound rather than network-bound.
6. **The real NVMe→NVMe workload confirms the synthetic curve** — same shape, same ceiling,
   well under the local storage ceilings.

### Honest caveats

- **`busy_cores` under-counts.** It counts cores >90% utilized; with work spread across 192
  cores few individual cores cross that bar, so `busy_cores` often reads 0 and the derived
  `Gbps/busy-core` column is noisy. `max_core_util` is the reliable saturation signal and is
  what drives the CPU/crypto attribution.
- **The single saturated receive core was not isolated to a specific kernel thread**
  (ksoftirqd vs the nvme/iperf worker). The measured facts are: one core pegged at ~99% on
  the receiver, no allowance fired, throughput flat. Pinning down that core is the obvious
  next investigation.
- **Short windows (15 s).** Long enough for steady state on the sweep, but SRD's adaptive
  ramp wants longer; a multi-minute `ena_express` run might engage SRD materially.
- **DUR=15, N grid 1–32.** A finer grid near the N=8 knee and higher N would sharpen the
  plateau characterization.

### What would push past 60 Gbps (hypotheses, not yet measured)

Raising the per-core/aggregate crypto rate is the lever: more receive queues actively
spreading softirq, RPS/RFS tuning, confirming all 32 wg interfaces' RX truly land on distinct
cores, or offloading the decrypt path. The network and storage both have ample headroom
(208 Gbps and 52 GB/s); only the encrypted-path CPU work binds.

### Follow-up instrumentation (added after this run; ready for the next)

The first run could see *that* one core pegged at ~99% but not *which thread* or *how many
cores' worth* of crypto was actually running — `collect.sh` only recorded "cores >90%" (which
read 0 when work spread thinly) and a single max. That has been addressed:

- **`collect.sh`** now emits a per-core utilisation histogram, **busy core-equivalents**
  (`sum((100-idle)/100)` = how many full cores' worth of work the binding side spent), and a
  **`top_threads`** sample (via `pidstat -t`) naming the hottest threads during the window —
  so a future run will show e.g. `ksoftirqd/NN:99` vs `iperf3:NN` and settle whether the
  pegged core is softirq/decrypt or userspace receive.
- **`report`** surfaces `core-equiv (A/B)`, `Gbps/core` (now off core-equivalents, not the
  noisy >90% count), and a **Hot threads** table.
- **`pin-workers.sh`** pins each tunnel's iperf3 worker to a distinct core, as an A/B knob to
  isolate how much of the ceiling is CPU-scheduler float vs raw per-core ChaCha20 throughput.

Re-running the matrix with this instrumentation is a fresh ~$5 Spot session; the open question
("how many cores decrypt, and is the pegged core softirq or userspace") is then directly
answered from the data rather than inferred.
